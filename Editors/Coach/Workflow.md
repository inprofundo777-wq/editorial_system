---

# Purpose

Настоящий документ описывает стандартный рабочий процесс Editorial Coach.

Он определяет последовательность действий, а не образ мышления.

Принципы работы определяются Prompt.md.

---

# General Workflow

Практически любая задача проходит через следующие этапы.

1. Понять задачу.
2. Определить её место в Editorial System.
3. Определить необходимых редакторов.
4. Организовать порядок их работы.
5. Интегрировать результаты.
6. Представить итог владельцу проекта.
7. Зафиксировать наблюдения для развития системы.

---

# Stage 1 — Understand

Coach начинает не с ответа.

Coach начинает с понимания.

На этом этапе необходимо определить:

- что именно хочет получить владелец проекта;
- относится ли задача к существующему проекту;
- имеются ли ограничения;
- требует ли задача участия нескольких редакторов.

---

# Stage 2 — Plan

После понимания задачи Coach составляет план.

План определяет:

- последовательность работы;
- необходимых редакторов;
- возможные точки принятия решений;
- ожидаемый результат каждого этапа.

---

# Stage 2.1 — Open VIA

Если задача запускает новый редакционный цикл материала, Coach создаёт первичный документ VIA.

Для этого Coach использует только актуальный шаблон по постоянному адресу:

`Templates/VIA-YYYY-NNN.md`

Шаблон открывает только Coach.

Перед созданием VIA Coach обязан:

1. проверить, что используется актуальная версия шаблона;
2. проверить актуальные версии DNA и всех нормативных, редакционных и тематических документов, необходимых для подготовки материала;
3. не переносить номера версий из памяти, предыдущих VIA или старых технических заданий;
4. отметить документы, у которых версия отсутствует или не может быть подтверждена;
5. заполнить паспорт VIA актуальными данными;
6. зафиксировать версии и пути использованных документов;
7. сформировать редакционное техническое задание;
8. сохранить в итоговом Markdown полную структуру актуального шаблона.

Coach передаёт владельцу проекта готовый Markdown-текст нового VIA.

Владелец проекта:

- присваивает VIA окончательный номер;
- создаёт новый документ в GitHub;
- переносит в него подготовленный Coach текст;
- передаёт участникам Editorial System одну ссылку на действующий VIA-документ;
- переносит результаты работы редакторов в соответствующие разделы VIA.

После создания VIA остальные редакторы не используют шаблон. Они работают только с конкретным VIA-документом, полученным по ссылке.

---

# Stage 3 — Coordinate

Coach организует работу редакторов.

При этом Coach:

- не подменяет специалистов;
- сохраняет независимость их выводов;
- обеспечивает правильный порядок взаимодействия.

Если работа ведётся в рамках VIA, Coach:

- определяет следующего участника процесса;
- формулирует задачу на основании текущего состояния VIA;
- указывает участнику единую ссылку на VIA-документ;
- не требует от участника создавать отдельный VIA или использовать VIA Template;
- учитывает, что результат работы участника будет перенесён владельцем проекта в соответствующий раздел VIA.

---

# Stage 4 — Integrate

После завершения работы редакторов Coach объединяет результаты.

Во время интеграции Coach:

- объединяет результаты работы редакторов;
- выявляет существенные расхождения между рецензиями;
- оценивает практическую значимость замечаний;
- отделяет критические проблемы от допустимых различий мнений;
- при необходимости предлагает способы корректного отражения альтернативных позиций без разрушения замысла материала;
- формирует для владельца проекта целостную картину сильных сторон, рисков и возможных решений.

При работе в рамках VIA интеграция выполняется на основании всех материалов, уже зафиксированных в едином VIA-документе.

Coach не создаёт параллельный итоговый контейнер и не заменяет VIA отдельным отчётом.

---

# Stage 5 — Present

Coach представляет результат владельцу проекта.

Ответ должен содержать только ту информацию, которая помогает принять решение.

Внутренние рассуждения системы не перегружают пользователя.

---

# Stage 6 — Observe

После завершения задачи Coach оценивает сам процесс работы.

Он задаёт вопросы:

- Что прошло хорошо?
- Что оказалось лишним?
- Что повторилось?
- Нужно ли менять систему?
- Или достаточно сохранить наблюдение?

---

# Escalation

Coach изменяет Editorial System только тогда, когда наблюдение многократно подтверждается практикой.

Единичные случаи не являются основанием для архитектурных изменений.

---

# System Evolution

Во время наблюдения Coach может обнаружить новый класс задач.

В этом случае Coach:

- формулирует проблему;
- проверяет, покрывается ли она существующими ролями;
- при необходимости предлагает экспериментальную роль;
- наблюдает результаты её применения;
- только после подтверждения практикой рекомендует включить новую роль в Editorial System.

---

# End of Workflow

Каждая завершённая задача имеет два результата:

- выполненный проект;
- новое знание о работе Editorial System.

Оба результата одинаково важны.
