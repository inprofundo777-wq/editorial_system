# Constitution

Папка `Constitution` содержит документы, определяющие природу In Profundo, архитектуру Editorial System, первый цикл формирования нормативного корпуса и связанные с ним документы памяти и развития.

Настоящий `README.md` является навигационным и статусным документом папки. Он не заменяет содержательные документы и не обладает самостоятельной нормативной силой.

---

# 1. Визуальная система статусов

Для быстрого понимания состояния документа используется принцип светофора.

| Маркер | Статус | Значение |
|---|---|---|
| 🟢 | Готов / действует | Документ утверждён, принят или завершён для текущего этапа. Может использоваться как источник истины в пределах своего статуса. |
| 🟡 | В работе | Документ обсуждается, пишется, проходит review или нуждается в доработке. |
| 🔴 | Требует разработки | Документ находится на ранней стадии: требуется исследование, обсуждение, архитектурное решение или подготовка первого текста. |
| ⚪ | Заготовка / не определён | Файл существует технически, но не является документом и не имеет утверждённой функции. |
| ⚫ | Архив | Документ сохраняется как исторический источник, но не регулирует текущую работу. |

Цветовой маркер всегда дополняет, но не заменяет текстовый статус.

---

# 2. Основной принцип работы с документами

Работа над постоянными документами проходит по следующему циклу:

1. документ обсуждается и разрабатывается в профильной рабочей ветке;
2. после согласования создаётся полный итоговый файл в формате Markdown;
3. файл переносится в GitHub;
4. ссылка возвращается для проверки;
5. после проверки GitHub-файл становится действующим источником истины;
6. дальнейшая работа ведётся по GitHub-версии, а не по истории чата;
7. при выпуске новой редакции прежняя версия сохраняется в архиве с точным номером версии;
8. постоянное имя действующего документа не меняется, чтобы не нарушать внутренние ссылки.

Пустой файл, заготовка или документ без указанного статуса не является нормативным документом.

# 6. Периодический review статусов

Статусы папки пересматриваются:

- после завершения каждого крупного документа;
- после Foundational Review или иного фундаментального review;
- при изменении очередности работ;
- при создании нового документа;
- при архивировании или замене действующего текста;
- не реже одного раза в три месяца в период активного развития системы.

При review необходимо:

1. проверить фактический статус каждого файла;
2. обновить цветовой маркер;
3. зафиксировать ближайшее действие;
4. удалить ложные или устаревшие статусы;
5. проверить, не воспринимается ли пустая заготовка как действующий документ.
